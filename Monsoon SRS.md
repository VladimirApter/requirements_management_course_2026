## 1. Введение

### 1.1. Назначение документа

* **Цель документа**: описать функциональные требования для системы Monsoon с указанием приоритета для разработки:
  * **H** — высокий: оказывает высокое влияние на работоспособность MVP
  * **M** — средний: оказывает среднее влияние на работоспособность MVP
  * **L** — низкий: оказывает низкое влияние на работоспособность MVP

* **Название системы**: Monsoon

### 1.2. Рамки и цели создания системы

Система будет создаваться в несколько этапов. На первом этапе будет создан MVP.

Данный документ описывает требования к MVP системы.

После завершения разработки MVP должен соответствовать следующим пользовательским метрикам:

* **Метрика №1:** Время публикации работы художником
  * Целевое значение: менее 2 минут от начала заполнения до завершения загрузки
  * Способ измерения: аналитика действий в модуле публикации

* **Метрика №2:** Процент художников, вернувшихся для загрузки 3 и более работ
  * Целевое значение: более 40% от активных художников
  * Способ измерения: cohort analysis по данным БД

* **Метрика №3:** Конверсия из просмотра работы в заявку на покупку
  * Целевое значение: более 1% от просмотров
  * Способ измерения: отношение количества заявок к количеству просмотров работ

* **Метрика №4:** Время модерации одной работы
  * Целевое значение: менее 3 минут
  * Способ измерения: время между статусами "на модерации" и "одобрена"

Система должна обеспечить получение следующих бизнес-преимуществ:

* **Бизнес-преимущество 1:** Снижение операционной нагрузки на команду
  * Автоматизация модерации позволит обрабатывать работы за <3 минут, что масштабируется на рост количества художников

* **Бизнес-преимущество 2:** Формирование сетевого эффекта без больших затрат
  * Социальные механики (лайки, подписки, коллекции) создают привычные паттерны поведения, снижая барьер перехода с Instagram

* **Бизнес-преимущество 3:** Дифференциация от конкурентов
  * Упрощенная публикация в 2 шага против 15 полей на ArtNow — ключевое конкурентное преимущество для художников

### 1.3. Термины и сокращения

* **Работа (Artwork):** цифровое представление физической картины, загруженное художником, включающее изображение и метаданные (название, жанр, размер, цена, описание)
* **Коллекция (Collection):** набор работ, сохраненных зрителем в виртуальную папку для последующего просмотра
* **Заявка на покупку (Purchase Request):** запрос от зрителя на приобретение работы
* **Модерация (Moderation):** процесс проверки загруженной работы на соответствие правилам платформы перед публикацией
* **MVP (Minimum Viable Product):** минимально жизнеспособная версия продукта с базовым набором функций
* **RBAC (Role-Based Access Control):** управление доступом на основе ролей

## 2. Общее описание

### 2.1. Взаимодействие с системой

Рис.1 Контекстная диаграмма

#### 2.1.1. Роли и классы пользователей

Система должна предоставлять возможности для автоматизации процессов следующих классов пользователей:

* **Художник** — пользователь, который публикует свои работы, управляет профилем, получает обратную связь (лайки, подписки, сохранения в коллекции) и заявки на покупку через администратора
* **Зритель** — пользователь, который просматривает ленту работ, ставит лайки, подписывается на художников, создает коллекции и отправляет заявки на покупку
* **Модератор контента** — административная роль, проверяющая загружаемые работы на соответствие правилам перед публикацией
* **Администратор заявок** — административная роль, обрабатывающая входящие заявки на покупку, связывающая художника и зрителя

#### 2.1.2. Вспомогательные подсистемы

Система должна осуществлять взаимодействие со следующими подсистемами:

1. **Подсистема аутентификации (Auth Service)**
  * **Назначение:** обеспечение регистрации и входа пользователей
  * **Взаимодействие:** интеграция через API для регистрации по email/паролю и через внешних провайдеров (Google, VK ID)

2. **Подсистема уведомлений (Notification Service)**
  * **Назначение:** доставка уведомлений пользователям на сайте
  * **Взаимодействие:** внутренняя подсистема для создания и отображения уведомлений в центре уведомлений

#### 2.1.3. Смежные системы (источники информации)

1. **Облачное хранилище изображений (Cloudinary / S3)**
  * **Назначение:** хранение оригиналов и оптимизированных версий работ, CDN-доставка
  * **Взаимодействие:** API для загрузки изображений при публикации работ и получения URL для отображения

2. **Система аналитики (Яндекс.Метрика / Google Analytics)**
  * **Назначение:** сбор метрик поведения пользователей
  * **Взаимодействие:** встраиваемый код для отправки событий (просмотры страниц, клики, лайки, сохранения, отправка заявок)

#### 2.1.4. Потоки данных

* **Приемник ← Источник**
  * **Художник → Модуль публикации работ:** изображение работы, название, жанр, размер, цена, описание
  * **Зритель → Модуль заявок на покупку:** выбранная работа, контактные данные, сообщение
  * **Модератор → Модуль модерации:** статус модерации (одобрена/отклонена), комментарий

* **Источник → Приемник**
  * **Модуль публикации работ → Облачное хранилище:** оригинальное изображение работы
  * **Модуль модерации → Модуль уведомлений:** статус модерации работы для уведомления художника
  * **Модуль заявок на покупку → Модуль уведомлений:** статус обработки заявки для уведомления зрителя
  * **Модуль социального взаимодействия → Модуль уведомлений:** лайк, подписка, сохранение в коллекцию для уведомления художника

### 2.2. Ограничения к ПО

#### 2.2.1. Ограничения к операционной среде

* **ОС-1:** Веб-приложение должно корректно работать в браузерах последних 2 версий: Chrome, Firefox, Safari, Edge
* **ОС-2:** Серверная часть должна функционировать под управлением Linux (Ubuntu 20.04 или выше)
* **ОС-3:** Поддержка только веб-версии; мобильные приложения не входят в границы MVP

#### 2.2.2. Ограничения проектирования и реализации

* **ОПР-1:** В качестве СУБД должна использоваться PostgreSQL версии 14 или выше
* **ОПР-2:** Бэкенд-логика должна быть реализована на Python с использованием фреймворка FastAPI
* **ОПР-3:** Фронтенд должен быть реализован с использованием React.js или Vue.js
* **ОПР-4:** Интеграция посредством API с внешними системами (Cloudinary, Google, VK ID)
* **ОПР-5:** Архитектура должна поддерживать RBAC (Role-Based Access Control) в соответствии с ролями: Художник, Зритель, Модератор контента, Администратор заявок

#### 2.2.3. Ограничения к оборудованию

* **ПО-1:** Серверное оборудование — облачный хостинг (Yandex Cloud / AWS) с минимальной конфигурацией: 2 vCPU, 4GB RAM, 50GB SSD

#### 2.2.4. Ограничения к сетевому соединению

* **СС-1:** API внешних систем (Cloudinary, Google, VK ID) должны быть доступны с временем ответа не более 2 секунд
* **CC-2:** Загрузка изображений должна выполняться по HTTPS с поддержкой chunked upload для файлов до 50MB

### 2.3. Требования к составу поставки

* **СП-1: Состав поставки**:
  * Исходный код системы (бэкенд + фронтенд)
  * Скрипты для развертывания (Docker Compose / Kubernetes manifests)
  * Миграции базы данных
  * Конфигурационные файлы для staging и production окружений

* **СП-2: Комплект документации к системе должен включать**
  * **Д-1:** Руководство администратора — описание установки, настройки RBAC, управления пользователями, резервного копирования
  * **Д-2:** Руководство пользователя — инструкции для художников, зрителей и модераторов
  * **Д-3:** Документ архитектуры — схемы компонентов, описание API, диаграммы потоков данных, модель БД
  * **Д-4:** Руководство программиста — требования к среде разработки, структура кода, соглашения по стилю, инструкции по сборке и тестированию

* **СП-3: Требования к лицензированию и патентной чистоте должны включать:**
  * **ЛПЧ-1:** Использование только open-source библиотек с лицензиями, совместимыми с MIT/BSD
  * **ЛПЧ-2:** Отсутствие использования проприетарных компонентов, требующих лицензирования
  * **ЛПЧ-3:** Все изображения, загружаемые пользователями, остаются их интеллектуальной собственностью
  * **ЛПЧ-4:** Платформа не претендует на права на контент пользователей
  * **ЛПЧ-5:** Использование стандартных шрифтов и иконок с открытыми лицензиями

### 2.4. Допущения и зависимости

В рамках MVP не предполагается:
* Встроенный чат между художником и зрителем
* Платежный шлюз (деньги проходят вне системы)
* Автоматическая email-рассылка уведомлений (уведомления только на сайте)
* Мобильные приложения (только веб-версия)
* Визуальный поиск по изображению

### 2.5. Сроки и бюджет

**2.5.1. Сроки и бюджетные ограничения**

* Бюджет на разработку MVP: 300 000 рублей (собственные средства команды)
* Команда разработки: 5 человек (все необходимые роли закрыты)
* Срок MVP: 3 месяца

**2.5.2. Стратегия внедрения MVP**

* **Пилот:** Привлечение 10-20 художников для тестирования функционала публикации работ
* **Тестирование:** Прототип передаётся первым пользователям, команда отслеживает поведенческие метрики (возвраты, просмотры, использование поиска и рекомендаций) и собирает данные для улучшения продукта
* **Критерий успеха:** Не менее 40% привлеченных художников возвращаются и загружают более 3 работ; время публикации работы менее 2 минут

### 4.3. Детальное описание способов применения по модулям

#### 4.3.1. Модуль 2: Публикация работы

* **UC-M2P1.1: Опубликовать новую работу**
  * **Краткое описание:** Художник загружает изображение своей картины и заполняет информацию о работе в упрощенной форме из 2 шагов
  * **Акторы:** Художник
  * **Предусловия:**
    - Художник авторизован в системе
    - Художник имеет активный профиль
  * **Входные данные:**
    - Изображение работы (jpg, png, webp, до 20MB)
    - Название работы
    - Жанр (из предопределенного списка)
    - Размер (высота и ширина в см)
    - Цена (в рублях)
    - Описание (до 2000 символов)
  * **Связь с функциональными требованиями:** FR-M2P1.1; FR-M2P1.2; FR-M2P1.4; FR-M2P1.5; FR-M2P1.6; FR-M2P1.7
  * **Основной ход (Main Flow):**
    1. Художник нажимает кнопку "Опубликовать работу" в интерфейсе
    2. Система открывает шаг 1 формы: загрузка изображения
    3. Художник выбирает файл изображения на устройстве
    4. Система отображает предварительный просмотр изображения и кнопку "Далее"
    5. Художник нажимает "Далее", система переходит к шагу 2
    6. Система отображает форму с полями: название, жанр, размер (высота/ширина), цена, описание
    7. Художник заполняет все обязательные поля
    8. Художник нажимает кнопку "Опубликовать"
    9. Система сохраняет изображение в облачном хранилище, генерирует превью
    10. Система сохраняет данные работы в БД со статусом "на модерации"
    11. Система отображает сообщение об успешной публикации и направляет художника в профиль
  * **Альтернативные потоки (Extensions):**
    * **1-A1: Неподдерживаемый формат файла**
      - 4-A1.1: Система проверяет формат файла
      - 4-A1.2: Если формат не поддерживается (не jpg/png/webp), система показывает сообщение об ошибке
      - 4-A1.3: Возврат к шагу 4 основного потока
    * **1-A2: Превышение размера файла**
      - 4-A2.1: Система проверяет размер файла
      - 4-A2.2: Если размер превышает 20MB, система показывает сообщение об ошибке
      - 4-A2.3: Возврат к шагу 4 основного потока
    * **8-A1: Не заполнены обязательные поля**
      - 8-A1.1: Система проверяет заполнение обязательных полей
      - 8-A1.2: Если поля не заполнены, система подсвечивает их красным и показывает сообщение
      - 8-A1.3: Возврат к шагу 8 основного потока
  * **Постусловия:**
    - Создана запись о работе в БД со статусом "на модерации"
    - Изображение загружено в облачное хранилище
    - Создана задача для модератора в очереди на проверку
  * **Связанные UC:** UC-M2P2.2 (Одобрить работу), UC-M2P2.3 (Отклонить работу)

#### 4.3.2. Модуль 3: Социальные действия

* **UC-M3P2.1: Поставить лайк работе**
  * **Краткое описание:** Зритель ставит лайк на понравившуюся работу
  * **Акторы:** Зритель (авторизованный)
  * **Предусловия:**
    - Зритель авторизован в системе
    - Работа опубликована и одобрена (статус "одобрена")
    - Зритель не ставил лайк этой работе ранее
  * **Входные данные:** ID работы
  * **Связь с функциональными требованиями:** FR-M3P2.1; FR-M3P2.3; FR-M3P2.11
  * **Основной ход (Main Flow):**
    1. Зритель видит работу в ленте или на детальной странице
    2. Зритель нажимает кнопку лайка (иконка сердца)

  * **Альтернативные потоки (Extensions):**
    * **2-A1: Повторный клик на лайк (отмена лайка)**
      - 2-A1.1: Если зритель нажимает на активную кнопку лайка, система уменьшает счетчик на 1
      - 2-A1.2: Система меняет состояние кнопки на неактивное
      - 2-A1.3: Система удаляет запись о лайке из БД
      - Завершение сценария
  * **Постусловия:**
    - Увеличено количество лайков работы на 1
    - Создана запись в таблице лайков
    - Создано уведомление для художника
  * **Связанные UC:** UC-M3P2.2 (Отменить лайк)

### 4.4. Требования к качеству использования

#### 4.4.1. Требования к результативности

* **РЗ-1:** Художник должен успешно публиковать работу за 2 минуты или менее в 95% случаев
* **РЗ-2:** Зритель должен успешно отправлять заявку на покупку за 1 минуту или менее в 95% случаев
* **РЗ-3:** Модератор должен успешно обрабатывать одну работу за 3 минуты или менее в 90% случаев

#### 4.4.2. Требования к скорости обучения

* **ОБ-1:** Новый художник должен успешно опубликовать первую работу без обращения в поддержку в 90% случаев
* **ОБ-2:** Новый зритель должен понять механизм лайков и подписок без инструкции в 95% случаев (интерфейс интуитивно понятен)
* **ОБ-3:** Новый модератор должен научиться работать с интерфейсом модерации за 10 минут

#### 4.4.3. Требования к удовлетворенности

* **УД-1:** 80% художников должны оценить процесс публикации как "простой" или "очень простой" (опрос после 5 публикаций)
* **УД-2:** 85% зрителей должны оценить ленту как "удобную для просмотра" (опрос)
* **УД-3:** 70% зрителей, отправивших заявку, должны подтвердить, что процесс был понятным

## 5. Атрибуты внешнего качества ПО

### 5.1. ВК-П Требования к производительности

* **ВК-П1:** Время загрузки ленты (первые 20 работ) не должно превышать 2 секунд при стандартном интернет-соединении (10 Мбит/с)
* **ВК-П2:** Время загрузки детальной страницы работы не должно превышать 1.5 секунд
* **ВК-П3:** Система должна поддерживать одновременную работу до 1000 пользователей без деградации производительности
* **ВК-П4:** Загрузка изображения должна выполняться с отображением прогресса; время загрузки файла 10MB не должно превышать 30 секунд
* **ВК-П5:** API должен обрабатывать до 50 запросов в секунду с временем ответа менее 200 мс (p95)

### 5.2. ВК-М Требования к масштабируемости

* **ВК-М1:** Архитектура должна позволять горизонтальное масштабирование бэкенда (добавление новых экземпляров приложения)
* **ВК-М2:** База данных должна поддерживать шардирование или репликацию для масштабирования чтения
* **ВК-М3:** Хранение изображений должно использовать облачное решение с автоматическим масштабированием

### 5.3. ВК-Н Требования к надежности

* **ВК-Н1:** Система должна обеспечивать время безотказной работы (uptime) не менее 99.5%
* **ВК-Н2:** В случае сбоя при загрузке изображения система должна корректно обрабатывать ошибку и позволять повторную загрузку без потери данных формы
* **ВК-Н3:** Данные пользователей должны сохраняться атомарно: либо вся операция выполнена, либо система возвращается к исходному состоянию

### 5.4. ВК-Б Требования к безопасности

* **ВК-Б1:** Пароли пользователей должны храниться в зашифрованном виде (bcrypt или PBKDF2)
* **ВК-Б2:** Все запросы к API должны выполняться по HTTPS; HTTP-запросы должны перенаправляться на HTTPS
* **ВК-Б3:** JWT-токены для аутентификации должны иметь время жизни не более 24 часов
* **ВК-Б4:** Доступ к административным функциям (модерация, обработка заявок) должен быть ограничен по RBAC
* **ВК-Б5:** API должен иметь защиту от CSRF-атак
* **ВК-Б6:** Должна быть реализована защита от brute-force атак на вход (ограничение количества попыток)

### 5.5. ВК-ОУ Требования к отказоустойчивости

* **ВК-ОУ1:** Система должна иметь резервное копирование базы данных не реже 1 раза в сутки
* **ВК-ОУ2:** При недоступности внешних сервисов (Cloudinary, Google OAuth) система должна корректно обрабатывать ошибки и показывать понятные сообщения пользователю
* **ВК-ОУ3:** В случае сбоя при отправке уведомления, система должна повторять попытку в течение 24 часов (retry-механизм)

### 2.6. Отсутствие трассировки / соответствие бизнес-целям

**Замечание #13**

## 3. Рекомендации по приоритетам

### P0 (критично, блокирует разработку)

1. **Замечание #11** — противоречие между лимитом загрузки (20MB) и поддержкой chunked upload до 50MB. Должно быть исправлено до начала проектирования API.
2. **Замечание #7** — нетестируемое требование к производительности (условия тестирования не определены). Без уточнения невозможно принять результат тестирования.

### P1 (значительно влияет на качество)

3. **Замечание #4, #5, #6** — неоднозначные формулировки требований к UX («интуитивно понятный», «простой», «Pinterest/TikTok стиль»). Влияют на согласование ожиданий с дизайнерами и фронтендом.
4. **Замечание #8** — отсутствие критериев деградации производительности при нагрузке.
5. **Замечание #13, #14** — требования без трассировки на бизнес-цели. Могут привести к реализации ненужных функций за ограниченный бюджет.
6. **Замечание #16** — отсутствие требований к логированию и мониторингу. Затруднит эксплуатацию и отладку на MVP.

### P2 (улучшаемо, не критично)

7. **Замечание #1, #2, #3** — неполнота сценариев (исключительные потоки). Рекомендуется для полноты документа, но не блокирует начало разработки.
8. **Замечание #10** — нарушение атомарности. Не критично, но может привести к неполной реализации функционала коллекций.
9. **Замечание #15** — избыточное требование группировки уведомлений. Может быть исключено из MVP без ущерба для метрик.
10. **Замечание #17, #18** — недостающие NFR (accessibility, RPO/RTO). Важны для production-релиза, но для MVP могут быть отложены.
